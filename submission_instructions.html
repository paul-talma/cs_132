CS132: Submission Requirements

All submissions to CS132 needs to be a single Tar file that follow a set of requirements:

  1. It MUST have your UID as name; example: 123456789.tar.

  2. It MUST contain ONLY and ALL the source code and used to compile your program. The files should be directly compressed into the tar. You should not compress the folder containing the tar files. Hence, the tar file structure should look like this:

    $ tar -vtf 123456789.tar
<main-class>.java
other/packages/Used.java
other/packages/My.java

  3. Its contents MUST be able to build, and get a valid score with the pregrade command (explained below).

  4. The content of the folder is tested using Java SE 1.8.0_101 ONLY. Features from Java 9 and up will NOT compile on the server and will result in a score of 0.

Notes for Windows Users: The commands described above should work on any of the UNIX SEAS machines that is available to you. If in doubt ask your TA.

Testing and feedback

To make everything easier for you we have created this automatic tester, packager, and pregrader. To use this helper tool, you can write your code in src/main/java, and any unit tests you want in src/test/java.

  1. First of download and install gradle.

  2. change MY_UID to your UID in gradle.properties, and the homework variable to the current homework (e.g., hw1).

  3. make sure your main class matches the expected main class for the assignment. See mainclass in build.gradle.

  4. build, and unit-test the code using gradle test.

You can run the code directly using the run task. Examples are available in the testcases folder. For example, to run against the first testcase for the first homework:

gradle run < testcases/hw1/test01

Finally to build your project and run against all publicly available tests simply run gradle pregrade. The pregrader is the grader that we will use to grade you, but more test-cases will be added in the final grading. Your score will be the ratio of test cases passed, except for HW-4 (Register Allocation) which is explained in detail below.

NOTE: The current pregrade script uses bash, so you might have some problems using Windows.

Javadoc (for HW 2 onwards)

Open up the file javadoc/index.html using a browser to access the Javadoc of the API. This gives you the list of AST classes and visitors available for you to use. For HW2, you will use cs132.minijar package entries, for HWs 3, 4 and 5 you will additionally use the cs132.IR packages. You will typically extend one of the available visitors and override the visit methods for each of the AST classes to translate them.

Using the Interpreters to debug (for HWs 3,4,5)

For these 3 HWs your code will perform a translation and output a program. The output will be in Sparrow for HW3, in SparrowV for HW4 and RiscV for HW5. To test if your output is a valid program, you can run the provided interpreter on your program output. Here is an example of how you could run the output of your program on the interpreter.


	gradle run < testcases/hw3/test1.java > out.sparrow
	java -jar misc/sparrow.jar s < out.sparrow

Before running the sparrow.jar on out.sparrow, remember to clear the extra lines printed by gradle at the beginning and end of out.sparrow.

Here are corresponding commands for SparrowV and RiscV


	java -jar misc/sparrow.jar sv < sparrowVcode.sparrowv
	java -jar misc/venus.jar < riscVcode.txt

Homework 4 (Register Allocation) Grading

This HW will have a different grading scheme. Simply passing all the test cases will give you only 20 percent of the grade (this is what will be tested by 'gradle pregrade' command). Hence getting a full score in 'gradle pregrade' will typically only give you a score close to 20 percent on the server. To get a higher score on the server, you need to optimize the number of loads+stores to the stack. This can be achieved by using some form of register allocation to keep the local and parameter variables in registers instead of storing to and loading from the stack. The fewer the loads/stores to the stack, the higher your grade will be. To get a full score, your loads+stores should be below a particular threshold.

Modifying the number of heap accesses from the original program will give a score of 0. If a test case crashes when executed by the interpreter, that test case will get a score of 0.

To measure the number of stack/heap accesses of the program output by your compiler run the following. 


	gradle run < testcases/hw4/test1.sparrow > out.sparrowv
	java -jar misc/sparrow.jar sv stats < out.sparrowv

Before running the sparrow.jar on out.sparrow, remember to clear the extra lines printed by gradle at the beginning and end of out.sparrowv

These stats will help you confirm whether your optimization/register-allocation actually reduced the number of loads/stores on the test program.

Running the Homework without Gradle (not recommended)

Create a folder to store the class files.

mkdir classes

To compile your code use either one of the following commands: the first command works only if you have no nested folders. The second command works for nested folders as well

javac -cp lib/cs132.jar -d classes src/main/java/*.java
javac -cp lib/cs132.jar -d classes $(find src/main/java/* | grep .java)

To test a single input

java -cp classes:lib/cs132.jar Typecheck < testcases/hw2/Basic-error.java

To generate your tar file

cd src/main/java
tar -czvf 000000000.tar *

To run pregrade

mv src/main/java/000000000.tar .
sh grade.sh hw2 . testcases/hw2 000000000.tar
